# Weelink improvement plans

Generated by a read-only `/improve` audit (4 parallel category sweeps: correctness/security,
performance/architecture, test coverage/DX/docs, dependencies/migrations/direction), personally
vetted finding-by-finding against the actual source before any plan was written. Every plan below
is self-contained — written for a separate executor with no memory of the audit conversation.

**Written against commit:** `6afe742`. Each plan re-states this; if the working tree has moved on
significantly before a plan is executed, re-verify the plan's "current state" excerpts against the
live file before trusting them (drift check).

**Audit scope disclosure:** this was a "standard" effort audit (hotspot-weighted, ≤4 concurrent
subagents). `apps/mobile` (Flutter) and a field-by-field review of the full Prisma schema were not
covered. The skill's reference files (`references/audit-playbook.md`,
`references/plan-template.md`) were not present in the installed skill package on this machine —
plans follow the structure specified inline in `SKILL.md` instead.

**Selection:** these 12 plans are the subset the user selected from the full audit findings table
(24 findings + 4 direction suggestions) — the cheapest, lowest-risk, highest-confidence fixes,
plus two higher-effort performance items the user judged worth doing now. Findings not selected
(major dependency bumps, Prisma migration-history retrofit, `pnpm lint` config, git hooks, refresh
token rotation, appointment race condition, admin XSS sanitization, unbounded queries, raw `<img>`
usage, blanket `"use client"`) were deliberately deferred — see the audit conversation for the
reasoning per item, not repeated here.

## Dependency graph

Almost all plans are independent and can be executed in any order. The two exceptions:

- **008 → nothing depends on it**, but its `.env.example` additions (`API_URL`, `WEB_URL`) document
  the same vars that **009** adds a boot-time hard requirement for. Recommend doing 008 before 009
  (docs before enforcement), though 009 does not technically require 008 to be merged first.
- **011 and 012 both edit `ShopsService.findBySlug`** in the same file
  (`apps/api/src/shops/shops.service.ts`). They are functionally independent (011 adds caching, 012
  adds new response fields) but will conflict if executed as parallel diffs against the same base
  commit. **Recommend executing 011 first, then 012** — 012's plan explicitly checks for and builds
  on top of 011's changes if present. If executing in the opposite order, 011's plan will need the
  same adjustment in reverse (cache the extra fields 012 adds too) — its plan doesn't currently
  assume 012 landed first, so re-verify before running 011 if 012 already merged.

Every other plan (001-007, 009, 010) touches disjoint files and can run in parallel with any other.

## Rotation / manual-action note

Plan 001 includes a manual credential-rotation step (Step 1) that requires checking live production
data and is explicitly marked "STOP and report back" if the executor isn't certain which account
to rotate — don't let an executor agent guess on this one.
